What we're building

Two GPU-backed inference services — YOLO for object detection and SAM for segmentation — deployed as SageMaker real-time endpoints. A FastAPI orchestrator running on ECS receives image requests, calls both endpoints, and returns merged JSON results to the client.

Why SageMaker

SageMaker manages the GPU infrastructure so the team focuses on model code, not EC2 cluster operations. Key benefits for this workload:

Managed GPU provisioning Built-in auto-scaling Model versioning via registry One-line rollback CloudWatch metrics included Multi-model endpoints (cost save)

Deployment process — 6 steps

1
Containerise each model
Each model (YOLO and SAM) gets its own Docker image with all dependencies. The container must expose a /invocations endpoint — SageMaker's required contract.
# yolo/Dockerfile
FROM pytorch/pytorch:2.1.0-cuda11.8-cudnn8-runtime
RUN pip install ultralytics fastapi uvicorn
COPY app.py .
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8080"]
Both YOLO + SAM need their own image
2
Push images to ECR
Elastic Container Registry stores the Docker images. SageMaker pulls from here at endpoint creation time.
# authenticate + push
aws ecr get-login-password | docker login --username AWS \
  --password-stdin 123456789.dkr.ecr.ap-southeast-2.amazonaws.com

docker tag yolo-inference:latest \
  123456789.dkr.ecr.ap-southeast-2.amazonaws.com/yolo-inference:v1

docker push 123456789.dkr.ecr.ap-southeast-2.amazonaws.com/yolo-inference:v1
3
Upload model weights to S3
SageMaker downloads weights at endpoint startup from S3. Keep containers lightweight — no weights baked in.
aws s3 cp yolov8n.pt s3://cv-models-bucket/yolo/yolov8n.pt
aws s3 cp sam_vit_h.pth s3://cv-models-bucket/sam/sam_vit_h.pth
S3 URI passed as MODEL_DATA_URL env var at deploy time
4
Create SageMaker endpoints
One endpoint per model. SageMaker provisions the GPU instance, pulls the container, downloads weights, and starts the server. Takes 3–5 minutes per endpoint.
import sagemaker

# YOLO endpoint
yolo = sagemaker.Model(
    image_uri="123456789.dkr.ecr.ap-southeast-2.amazonaws.com/yolo:v1",
    model_data="s3://cv-models-bucket/yolo/yolov8n.pt",
    role=role
)
yolo.deploy(instance_type="ml.g4dn.xlarge", endpoint_name="yolo-v1")

# SAM endpoint
sam = sagemaker.Model(
    image_uri="123456789.dkr.ecr.ap-southeast-2.amazonaws.com/sam:v1",
    model_data="s3://cv-models-bucket/sam/sam_vit_h.pth",
    role=role
)
sam.deploy(instance_type="ml.g4dn.2xlarge", endpoint_name="sam-v1")
5
Deploy the orchestrator on ECS
The FastAPI orchestrator runs on ECS Fargate (no GPU). It calls the SageMaker endpoints via the boto3 runtime client, using IAM role authentication — no API keys in code.
import boto3, asyncio

runtime = boto3.client("sagemaker-runtime", region_name="ap-southeast-2")

async def infer(image_bytes):
    # YOLO first — get bounding boxes
    yolo_out = runtime.invoke_endpoint(
        EndpointName="yolo-v1",
        ContentType="application/json",
        Body=image_bytes
    )
    boxes = json.loads(yolo_out["Body"].read())["boxes"]

    # SAM next — uses YOLO boxes as prompts
    sam_out = runtime.invoke_endpoint(
        EndpointName="sam-v1",
        ContentType="application/json",
        Body=json.dumps({"image": image_b64, "boxes": boxes})
    )
    return {"detections": boxes, "masks": json.loads(sam_out["Body"].read())}
6
Configure auto-scaling
Attach scaling policies to each endpoint so instances scale out under load and scale in (to a minimum of 1) when idle. This avoids over-provisioning costs.
# register YOLO endpoint as scalable target
aas = boto3.client("application-autoscaling")
aas.register_scalable_target(
    ServiceNamespace="sagemaker",
    ResourceId="endpoint/yolo-v1/variant/AllTraffic",
    ScalableDimension="sagemaker:variant:DesiredInstanceCount",
    MinCapacity=1, MaxCapacity=4
)
# scale out when GPU utilisation > 70%
aas.put_scaling_policy(
    PolicyName="yolo-gpu-scale", PolicyType="TargetTrackingScaling",
    TargetTrackingScalingPolicyConfiguration={
        "TargetValue": 70.0,
        "PredefinedMetricSpecification": {
            "PredefinedMetricType": "SageMakerVariantInvocationsPerInstance"
        }
    }
)
Repeat for SAM endpoint · min 1, max 2

Security layers

Network

SageMaker endpoints are deployed inside a private VPC subnet with no public IP. They are only reachable from the orchestrator ECS task via VPC endpoints — traffic never leaves the AWS backbone.

Private VPC subnet No public endpoint VPC endpoints (PrivateLink) WAF on API Gateway

Identity & access

Each service assumes an IAM role with only the permissions it needs. The ECS task role can invoke SageMaker and read/write S3. No AWS credentials are stored in containers or environment variables.

# ECS task role — minimal permissions example
{
  "Effect": "Allow",
  "Action": [
    "sagemaker:InvokeEndpoint",
    "s3:GetObject",
    "s3:PutObject"
  ],
  "Resource": [
    "arn:aws:sagemaker:ap-southeast-2:*:endpoint/yolo-v1",
    "arn:aws:sagemaker:ap-southeast-2:*:endpoint/sam-v1",
    "arn:aws:s3:::cv-models-bucket/*"
  ]
}

Data at rest & in transit

S3 buckets enforce SSE-S3 encryption on all objects. Input images are written by clients via presigned URLs with 5-minute TTLs. All traffic between services is TLS — API Gateway enforces HTTPS, and VPC internal calls use TLS by default.

S3 SSE-S3 at rest TLS in transit Presigned URLs (5 min TTL) No raw credentials in code

Audit & observability

CloudTrail logs every AWS API call. CloudWatch alarms fire on abnormal invocation spikes, high error rates, or GPU utilisation anomalies. Model invocation logs are enabled on both SageMaker endpoints.

CloudTrail audit log CloudWatch alarms SageMaker invocation logs
